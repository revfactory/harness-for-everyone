**핵심 설계 원칙:**
1. **엣지 우선**: 채널 어댑터와 인증을 Cloudflare Workers에 두어 한국 latency(평균 30ms)를 확보하고 API 오리진 부하를 차단.
2. **오케스트레이터 분리**: Node.js 오케스트레이터는 **작업 분해·모델 라우팅·결과 조립**만 담당. 실제 LLM 호출은 모델 라우터가 수행.
3. **서브에이전트 격리**: 각 서브에이전트는 독립 Worker/Lambda에서 실행. 크래시 격리 + 병렬 fan-out.
4. **상태와 연산 분리**: 상태(메모리·첨부)는 DB/S3에, 연산은 무상태 함수에. 비용 계량이 용이.
5. **모델 벤더 중립**: OpenRouter를 2차 게이트웨이로 두어 Anthropic 장애 시 GPT-4o-mini/Gemini Flash로 degraded 모드 자동 전환.

### 2-2. 라우팅 결정 로직 (의사코드)

```javascript
// orchestrator/router.ts
function routeModel(task) {
  // Step 1: L0 분류기로 태스크 특성 추출 (30ms)
  const features = classifier.classify(task); // { complexity, toolsNeeded, pii, longContext }

  // Step 2: 규칙 기반 우선 필터
  if (features.pii || task.type === 'batch_night') return 'L4'; // 로컬 모델
  if (task.channel === 'voice' && features.complexity < 0.3) return 'L1'; // 음성은 짧게

  // Step 3: 복잡도 점수
  const score = features.complexity * 0.6
              + features.toolsNeeded * 0.25
              + (features.longContext ? 0.15 : 0);

  // Step 4: 티어별 상한 적용
  const tier = user.plan; // free | light | pro
  const cap = { free: 'L1', light: 'L2', pro: 'L3' }[tier];
  const level = score < 0.35 ? 'L1'
              : score < 0.7  ? 'L2'
              : 'L3';
  return minLevel(level, cap);
}
```

## 3. 컨텍스트·메모리 구현 — "화이트보드 + 포스트잇"

### 3-1. 은유 ↔ 기술 매핑

| 은유(concept-translator) | 기술 구현 | 저장소 | TTL |
|----|----|----|----|
| **주방 화이트보드(단기)** | 대화 컨텍스트 윈도우 + 작업 스크래치패드 | Redis (session:{uid}:wb) | 작업 종료 + 2h |
| **벽면 포스트잇(장기 기억)** | 요약 메모리 노트 + 사용자 선호 | PostgreSQL + pgvector | 영구(사용자 삭제까지) |
| **단골 노트(특정 대상 기억)** | 엔티티별 요약 카드 (고객·거래처) | PostgreSQL (entities 테이블) | 영구 |
| **어제의 업무일지** | 작업 실행 로그·결과 | S3 (append-only JSONL) | 90일(무료) / 365일(유료) |
| **비밀 금고(권한·토큰)** | OAuth 토큰, API 키 | AWS KMS 암호화 Secret | OAuth TTL |

### 3-2. 벡터 DB 선택 — **pgvector 채택**

| 후보 | 비용 | 운영 복잡도 | 선택 사유 |
|------|------|-----------|---------|
| **pgvector** | PostgreSQL 확장, 추가 비용 없음 | 낮음(단일 DB) | ✅ 메모리 규모(사용자당 <5MB)에 적합. ACID 트랜잭션 |
| Qdrant | 자체 호스팅 $50~200/월 | 중 | 10M 벡터 이상 시 유리. 현재 규모 과함 |
| Chroma | 무료(OSS) | 중 | 단일 노드 한계. 30만 MAU 시 샤딩 부담 |
| Pinecone | 월 $70~ | 낮음 | SaaS 락인. 한국 리전 없음 |

**pgvector 설계:**
- 사용자별 메모리는 `memories(user_id, content, embedding vector(1024), kind, ttl, created_at)`.
- 임베딩 모델: **Voyage-3-lite ($0.02/M토큰)** 또는 **Cohere embed-multilingual-v3 ($0.10/M)**. 한국어 품질은 Voyage가 우세(자체 평가). 무료 대안: bge-m3 자체 호스팅.
- 1작업당 평균 메모리 3개 검색·1개 저장 → 임베딩 호출 ~200토큰 = **$0.000008** (무시 가능).

### 4-2. 스킬 로딩 모델 — "레시피 카드" 동적 로드

**문제:** Claude Code처럼 스킬(`.md` + 스크립트)을 읽어 실행하는 구조를 서버에서 어떻게 구현하나.

**해결:**

```
스킬 = Manifest(JSON) + System Prompt(md) + Executable(JS/WASM 옵션)
```

1. **Manifest 레지스트리**: `skills/{kind}/{name}/manifest.json` — 입력 스키마, 필요 권한, 모델 힌트.
2. **Orchestrator 로드**: 태스크 → 관련 스킬 TOP-3 선별(임베딩 + 태그) → 매니페스트와 시스템 프롬프트를 프롬프트 캐시에 주입.
3. **실행 분기**: 스킬이 순수 프롬프트면 모델 호출로 끝. 스크립트면 별도 함수(Workers/Lambda)로 격리 실행.
4. **사용자 커스텀 스킬** (프로 티어): 웹 에디터에서 마크다운 + 파라미터만 입력 → 서버가 매니페스트 자동 생성. 스크립트 업로드는 차단(보안).

### 5-1. 카카오톡 채널 — **최중요 채널**

## 7. MAX 능력 재현 전략 — 5개 기법

MAX 요금제가 제공하는 **① 장시간 루프 ② 긴 컨텍스트 ③ 높은 병렬성 ④ 깊은 추론 ⑤ 제한 없는 사용량**을 저비용으로 흉내내는 방법.

### 7-1. **체크포인트 재개(Resumable Workflow)**
- Temporal Cloud 또는 자체 Redis Streams 기반 워크플로우. 모든 스텝이 결정론적 이벤트로 기록 → 모델 실패·타임아웃 시 정확한 지점부터 재개.
- 효과: 60분 자율 태스크도 "실패 시 전체 재호출"이 아닌 "실패한 스텝만 재호출" → 토큰 최대 80% 절감.
- 구현: LangGraph 또는 자체 JSON 이벤트 머신. **LangGraph Python 채택 권고** (MVP 속도).

### 7-2. **요약 압축(Hierarchical Summary)**
- 긴 컨텍스트를 3단계로 압축: 원본 → 문단 요약 → 섹션 요약 → 전체 요약. 필요한 깊이만 로드.
- 200페이지 문서를 Sonnet 한 번 호출로 보내면 $0.60. 3단 압축 후 호출하면 $0.08.
- 라이브러리: LlamaIndex `HierarchicalNodeParser` 또는 자체 구현.

### 7-3. **스텝별 모델 승급(Escalation Ladder)**
- 모든 작업은 일단 **Haiku로 시도** → 신뢰도 점수(self-evaluation 0~1) < 0.7이면 **Sonnet 재시도** → 그래도 < 0.8이면 **Sonnet+Thinking**.
- 실제 승급 비율 가정: 70% Haiku 종료, 25% Sonnet, 5% Thinking. MAX 월정액 무제한 사용과 **결과 품질은 유사**하되 원가 1/10.

### 7-4. **작업 분할 큐(Fan-out·Fan-in)**
- "이 이메일 100개 중 중요한 것 뽑아줘"를 병렬 50개 Haiku fan-out → 결과를 Sonnet fan-in 집계.
- 단일 Sonnet 호출 대비 원가 65% 절감, 지연 50% 단축.
- 구현: SQS FIFO + Worker pool.

### 7-4-B. **야간 배치 크레딧(신규, v1.4) — 헤비 유저 평탄화 추가 완화**

critical-reviewer 지적에 대응한 추가 완화 기법. 기존 하드 캡·L1 다운시프트는 사후 방어인 반면, 이 기법은 **사전 수요 분산**.

- 사용자가 "급하지 않은 주문"을 **야간 배치 큐**(당일 23시~익일 06시)에 넣도록 유도 → **20% 사용량 크레딧 환급**
- 모델은 야간에 Anthropic API의 **Batch Processing 50% 할인**(2026-04 정책 활용) + **Qwen 자체 호스팅 유휴 GPU** 활용 → 원가 ~-55%
- 피크 시간 자영업자 마감(21~23시) 부하 평탄화에도 기여 (9-4와 정합)
- 헤비 5% 중 "실제론 시급하지 않은" 유저 절반을 야간으로 이관 → 실시간 경로 원가 30% 절감 예상

### 7-4-C. **상위 플랜 유도 신호(v1.4)** — 구조적 분리

월 300회 이상을 2개월 연속 기록하는 사용자는 **"프로 맥스 ₩49,000/월, 월 500회, L3 thinking 무제한"** SKU로 자연 이관 제안. 프로 티어의 마진 구조를 지키면서 헤비 사용자도 서비스에서 떠나지 않게 함.

### 7-5. **사전 구축된 스킬 라이브러리 (지식 외재화)**
- MAX의 "모델 자체가 알아서 다 함"을 **정제된 스킬·프롬프트 템플릿**으로 대체. 자영업 정산, 프리랜서 견적, 돌봄 기록 등 **페르소나별 치트시트 100개 사전 탑재**.
- 사용자 발화 → 스킬 매칭(임베딩) → 스킬의 제약된 프롬프트로 실행 → 결과 품질 평준화.
- 효과: "빈 프롬프트에서 창발 시도"의 토큰 낭비 제거.

## 8. 보안·권한 — "열쇠 꾸러미" 구현

### 8-1. 열쇠(권한) 3단계 기술 매핑

| 은유 | 실제 권한 | 구현 |
|------|---------|------|
| ⚪ 열쇠 없음 | 읽기도 안 함 | Scope 부재 |
| 🔵 초록 열쇠 (읽기) | OAuth read-only scope | 최소 권한 OAuth. 카카오·네이버·구글 각각 읽기 전용 토큰 발급 |
| 🟡 노란 열쇠 (실행) | OAuth write + 사용자 2차 승인 | 쓰기 액션마다 앱 푸시 "이거 진짜 할까요?" 확인 |
| 🔴 빨간 열쇠 (결제·외부 발송) | 금융·공개 발송 | **미지원(정책 차단)** |

**금지 스코프:** 금융(계좌이체·증권), 의료(병원 API), 행정(전자정부 대리 로그인), 타인 카톡 대리. 모두 아키텍처 레벨에서 블랙리스트.

### 8-2. OAuth 대리 실행 플로우 (v1.4 재아키텍처, critical-reviewer Blocker 대응)

**배경:** critical-reviewer 05 문서가 지적한 OAuth 자동 실행 리스크(금전·외부 발송 악용, 음성피싱 우회, 손해배상). UX팀의 🔓 권한 재설계(MVP에서 금전·외부 발송 자동 실행 제거, 실행 직전 앱 푸시 2단계 확인)와 정합.

**핵심 원칙:**
1. **MVP에서 쓰기 액션의 "자동 실행" 완전 제거.** 모든 쓰기는 사용자 확인 후에만 수행.
2. **읽기 권한 vs 쓰기 권한 분리 발급.** OAuth scope를 최소화하고 쓰기 scope는 별도 동의 단계.
3. **쓰기 실행 직전 앱 푸시 2단계 확인** 아키텍처 컴포넌트 신설.
4. **FDS(Fraud Detection System)** 규칙 엔진을 쓰기 경로에 필수 통과.
5. **금전·행정·의료 데이터 직접 연동 금지** — 아키텍처 수준 차단.

**플로우 (v1.4):**

```
[연결 단계]
1. 사용자가 앱에서 "📅 구글 캘린더 연결" 탭
2. 권한 선택 UI — [🔍 읽기만] / [✏️ 읽기+쓰기]
   · 기본 선택 = 읽기만. 쓰기는 명시적 별도 선택 필요
3. 브라우저 → Google OAuth consent → 코드 반환
4. 코드 교환 → access_token + refresh_token
5. 토큰 AWS KMS envelope 암호화 → DB 저장 (평문 로그 절대 금지)
6. scope 메타데이터(read/write) + 발급 시각 + 사용자 동의 약관 버전 해시 저장

[읽기 태스크 실행 — 기존 방식]
R1. KMS Decrypt → 메모리 전용 → API 호출 → 즉시 폐기

[쓰기 태스크 실행 — v1.4 신규 2단계 확인]
W1. 오케스트레이터가 "발송/쓰기" 의도 감지 → 쓰기 스코프 토큰 보유 확인
W2. 실행 직전 "승인 요청 페이로드" 생성:
    { action: "send_email", target: "holder@kim-co.kr",
      body_preview: "거래처 인사말…", amount: null, recipient_count: 3 }
W3. 💨 즉시 "주문 카드"로 사용자에게 앱 푸시 / 카톡 알림톡 발송
    "🔔 '지원이'가 손대려 합니다: 거래처에 메일 3통 보내려고 해요.
     내용 미리보기를 확인하고 [✅ 보내기] / [✋ 멈춰요]"
W4. 사용자가 앱 딥링크 탭 → 앱 진입 → **미리보기 + 영향 범위 + [확인/취소]** 표시
W5. [확인] 시 KMS Decrypt → FDS 규칙 통과 확인 → API 호출
    [취소] 시 영구 거부, 이후 같은 작업 재발 방지 플래그
W6. 결과 저장 + 감사 로그 (누가·언제·무엇을·왜 실행) + 사용자 피드백 슬롯

[토큰 라이프사이클]
L1. 읽기 토큰: 90일 슬라이딩 TTL, 자동 refresh
L2. 쓰기 토큰: 30일 짧은 TTL, refresh 시 재동의 요구
L3. 사용자 1회 이상 미사용(30일) 시 자동 만료 알림 → 60일 후 삭제
L4. 해지 버튼(앱 열쇠 꾸러미)에서 즉시 토큰 revoke + 감사 로그 기록
```

**FDS(Fraud Detection System) 규칙 엔진:**

쓰기 액션은 실행 직전 FDS를 반드시 통과. 실패 시 자동 차단 + 사용자 확인.

| 규칙 ID | 차단 조건 | 사용자 메시지 |
|--------|---------|-------------|
| FDS-01 | 최근 24h 내 쓰기 액션 ≥20건 | "오늘 너무 많이 움직이고 있어요. 확인 후 계속할게요." |
| FDS-02 | 본문에 "계좌·송금·입금·비밀번호·OTP" 키워드 | **즉시 차단 + 금전 관련 도움 불가 안내** |
| FDS-03 | 수신자가 최근 7일 내 처음 등장한 신규 외부 연락처 + 긴급성 표현 | "낯선 분께 급하게 보내시는 것 같아요. 잠시 확인해요." |
| FDS-04 | 한 번에 ≥50명 일괄 발송 | "많은 분께 한꺼번에 보내려 하세요. 명단 한 번만 확인해 주세요." |
| FDS-05 | 본문에 URL + "여기 클릭·로그인·확인" 조합 (음성피싱·스미싱 패턴) | **즉시 차단 + 피싱 의심 경고** |
| FDS-06 | 세션 IP가 평소 국가와 다름 (해외 접속) | "평소와 다른 곳에서 로그인하셨네요. 본인 확인 후 계속할게요." (SMS 인증) |
| FDS-07 | 음성 세션에서 쓰기 액션 시도 | **앱으로 강제 승급**(§ 5-2 정책과 일치) |

**음성피싱/보이스피싱 악용 방어:**

1. **음성 채널에서는 쓰기 스코프 토큰 사용 불가** (하드 차단, §5-2)
2. 음성 명령이 "급하게 보내요·지금 당장·아무한테도 말하지 말고" 같은 **긴급·비밀 프레임** 포함 시 모델 경고 트리거 + 앱 승급 강제
3. **지인 사칭 탐지**: "아버지·엄마·아들·딸"이 언급된 신규 쓰기 요청은 FDS-03과 함께 **두 채널 교차 확인**(앱 푸시 + SMS) 요구
4. **금전 키워드(FDS-02)는 아예 처리 불가** — 사용자에게 "저는 돈 관련 일은 돕지 않아요. 은행 앱에서 직접 해주세요" 응답 고정

**감사 로그 추가 요건:**

- 쓰기 액션 감사 레코드에 **FDS 통과 여부·규칙 히트 로그·사용자 승인 딥링크 클릭 타임스탬프·디바이스 IP/핑거프린트** 포함
- 사용자가 "이건 내가 안 한 일이다" 신고 시 즉시 **해당 세션 모든 OAuth 토큰 revoke + 30일간 쓰기 스코프 비활성 + CS 에스컬레이션**
- S3 append-only + KMS 암호화 + 5년 보관(금융감독원 표준 준용, 적용 의무 없더라도 모범 준수)

**토큰 저장 위치 옵션:**
- 기본: **서버 KMS 암호화** (편의·동기화)
- 프로 "주권 모드" 옵션(M9+): **클라이언트 디바이스 SecureStore**만 저장, 서버는 암호화 파사드만. 엔터프라이즈·기관 플랜에서 활성.

### 8-3. PII 마스킹 파이프라인

모든 사용자 입력은 LLM 전송 **직전** 마스킹:
```
원문: "김미영 사장님 010-1234-5678 매출 350만원"
마스킹: "[이름1] 사장님 [전화1] 매출 [금액1]"
맵: { "[이름1]": "김미영", "[전화1]": "010-1234-5678", ... }
```
- LLM 응답 후 맵 역적용. **LLM은 평문 PII를 절대 보지 않음**.
- 구현: presidio-kr(한국어 패치) 또는 자체 regex + KLUE-NER 경량 모델.
- 예외: 사용자가 명시적으로 "이 메일을 네이버 계정으로 보내줘"처럼 구체 값 유지를 요청한 경우만 bypass.

**스코어링 기준 (화이트리스트 추가 심사):**

- 심사 프로세스: 개발팀 제안 → 법무 검토 → 운영 승인 → YAML 화이트리스트 등록 → 감사 로그 자동 수집

**MVP 스코프 우선순위:** 1~6(필수) + 8·10·11(공공·결제 기반) = 9개만 MVP 런칭. 7(당근)·12(스마트플레이스)는 제휴 병행, 실패 시 M6까지 수동 UI 대체.

**기술 구현:**
1. 웹 자동화 스킬 실행 전 **도메인 화이트리스트 체크**. 비허용 도메인은 실행 거부.
2. 사용자 요청 "이 쇼핑몰 가격 비교해줘"는 **공식 API(네이버 쇼핑) 경로로만** 처리. DOM 크롤 불가.
3. 차단 상황 시 사용자에게 "이 사이트는 저희가 직접 못 들어가요. 사장님이 직접 하신 결과를 붙여넣어 주세요" 메시지 + 파일 업로드 UI.
4. 화이트리스트는 YAML 설정으로 운영자가 추가·제거. 법무 검토 거친 항목만 등록.
5. M6 이후 제휴 완료 파트너는 화이트리스트 확장(예: 스마트플레이스 공식 파트너 API 채택).

### 8-8. 소비자법 환불·약관 아키텍처 (v1.4 신설, critical-reviewer Q4 대응)

**배경:** critical-reviewer Q4 지적. "30일 100% 환불" 카피가 전자상거래법 제17조(청약철회)·표시광고법과 충돌 소지. 이에 기술·약관·UI를 정합시킨 환불 아키텍처 신설.

**법적 근거 매핑:**

| 법 조항 | 요구 | 우리 구현 |
|--------|-----|---------|
| 전자상거래법 제17조 제1항 | 7일 무조건 청약철회 (사용 이력 무관) | **"가입 후 7일, 사용 10회 미만: 자동 100% 환불"** |
| 전자상거래법 제17조 제2항 5호 | 디지털 콘텐츠 제공 시작 후 철회 제한 가능. 단 **사전 고지 의무** | 가입 Step 5에서 "사용 시작 시 청약철회 제한" 명시 + 체크박스 동의 필수 |
| 표시광고법 | 하드 캡 정책 고지 누락 시 허위·기만광고 | **가입 직전·결제 직전 하드 캡 고지 팝업** 의무 노출 |
| 방문판매법 | 자동결제 갱신 14일 전 통지 | **다음 결제 14일 전 알림톡** 자동 발송 |

**환불 규칙 엔진:**

```
사용자 환불 요청 → RefundPolicyEngine:
  CASE A: 가입 후 7일 이내 AND 사용 10회 미만
    → ✅ 즉시 자동 100% 환불 (전자상거래법 제17조)
  CASE B: 가입 후 7일 이내 AND 사용 10회 이상
    → ⚠️ 50% 환불 (프로레이타 + 서비스 비용 공제)
  CASE C: 가입 후 7~30일 AND 월 사용 10회 미만
    → ✅ 자동 100% 환불 ("실사용 거의 없는 이탈" 프렌들리 정책)
  CASE D: 가입 후 7~30일 AND 월 사용 10회 이상
    → 📋 CS 수동 검토. 기본 거부 + 품질 불만 확인 시 1회 한정 수동 환불
  CASE E: 30일 초과
    → ❌ 기본 거부. 중대 장애·서비스 하자 시 CS 개별 판단
```

**약관 버전 관리:**
- 약관 변경 시 새 버전 해시 + 사용자 재동의. User 테이블 `terms_version_hash` + `agreed_at`.
- 중대 불리 변경(가격 인상·주요 기능 제거) 시 **30일 사전 통지** + **무조건 해지·환불권** 부여.

**UI 통합 (UX팀 협의 예정):**
- 가입 Step 5: "7일 내 환불" + "하드 캡 한도 고지" + "자동 갱신 14일 전 알림" 3점 동시 노출 + 동의 체크
- 결제 직전 모달: "이번 달 199,000원 결제됩니다. 한도: 200회/월. 초과 시 건당 100원, 월 5만원 상한." 반드시 노출
- 홈 화면 상단 우측 항시 **[해지/환불] 진입점**. 3탭 이내 도달 가능
- 하드 캡 도달 시 **알림톡 + 앱 푸시** 이중 고지, 사용자 동의 없는 추가 과금 절대 금지

**감사 대비:**
- 동의·결제·환불 이벤트는 S3 append-only + KMS 암호화 5년 보관
- 공정거래위원회·한국소비자원 분쟁 조정 대응용 사본 생성 API

### 8-6. 침해 방어

- WAF (Cloudflare), Rate Limit (사용자당 10 req/s)
- Prompt Injection 방어: 사용자 입력은 항상 `<user_input>` 태그 래핑, 시스템 지시 분리. 외부 웹 페이지 내용은 별도 스코프 모델로 1차 필터.
- 모델 jailbreak 모니터링: 응답에 민감 키워드 탐지 시 플래그.

### 9-4. 피크 큐잉과 UX 연결

피크 시간(자영업자 마감 후 21~23시) 처리량이 2.3배. **R10 지적: AWS 서울 피크와 일치해 Fargate Spot 회수율 상승**. 방어:
1. **대기열 시각화** — "앞에 알바 대기 3명. 1분 내 시작" (UX 팀 화면 4와 연동)
2. **우선순위 큐** — 프로 > 라이트 > 무료
3. **자동 다운시프트** — 피크 중 L2 요청의 20%를 L1으로 강등(사용자 고지)
4. **배치 지연 옵션** — "10분 뒤 결과 받아도 되면 20% 할인 크레딧" 프로모션으로 평탄화
5. **Fargate 피크 대응(R10)** — On-demand 베이스라인 20% 상시 유지 + 피크 2시간 Spot 비중을 80%→50%로 자동 조정. 월 +$80 비용 수용. 최후 폴백은 **Lambda + Playwright 경량화**(cold start 5초 감수)
6. **피크 알림 사전 공지** — "저녁 9시 이후엔 살짝 느릴 수 있어요" UX 고지로 기대치 관리

ISMS-P + CSAP(하) 확보 시 공공 조달에서 **비인증 경쟁사 입찰 자격 자체 박탈**. 1회 투자 $75k가 2~3년 경쟁 장벽. IR 내러티브 자산.

### 11-5. 개인 시장 GM 버퍼와의 결합

11-3 비관/베이스 적자분은 MAU 10만+ 개인 시장 GM 89%(하이브리드 a70+c30) 버퍼에서 흡수. 월 $30k 절감분 중 **$10~15k/월이 기관 플랜 1~3년차 보조 가능**. 단 **무한 보조는 불가** — M24 철수 게이트 명문화.

### 11-6. 음성 데이터 정책 (UX 02 Y-6 / V-4 단일 진실 소스)

기술 구현은 § 5-2·8-5와 일치. UX 최종 카피는 `02_product_ux_designer_experience.md` Y-6·V-4에서 관리되며 본 문서는 참조만 둡니다.
